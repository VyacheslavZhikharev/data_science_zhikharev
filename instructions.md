 # Инструкция по работе с проектом

Этот проект прогнозирует свойства композитных материалов и сравнивает несколько подходов: классические модели, нейросети и кластеризацию.

## Что находится в проекте

- `hw_data_composite/X_bp.xlsx` — основная часть признаков.
- `hw_data_composite/X_nup.xlsx` — дополнительные признаки по нашивке.
- `1_ВКР_Жихарев_Вячеслав_Сергеевич_Свойства_композитов_Разведочный_анализ_и_Предобработка.ipynb` — загрузка, объединение и очистка данных.
- `2_ВКР_Жихарев_Вячеслав_Сергеевич_Свойства_композитов_Модели.ipynb` — классические модели регрессии.
- `3_ВКР_Жихарев_Вячеслав_Сергеевич_Свойства_композитов_Нейросеть.ipynb` — нейросетевые модели.
- `4_ВКР_Жихарев_Вячеслав_Сергеевич_Свойства_композитов_Кластеризация.ipynb` — кластеризация и моделирование по кластерам.
- `app.py` — CLI-приложение для прогноза целевой переменной.
- `requirements.txt` — список библиотек для установки.
- `input_example.json` — пример входных данных для CLI-приложения.

## Как устроен пайплайн (простыми словами)

1. Берем два Excel-файла из `hw_data_composite`.
2. Объединяем их по индексу (`INNER JOIN`) в одну таблицу `df`.
   После объединения автоматически сохраняем файл `Dataset_composites_inner.csv`.
3. Смотрим распределения, корреляции и качество данных.
4. Убираем явные ошибки (например, строку с нулями в ключевых признаках).
5. Нормализуем данные для части экспериментов.
6. Обучаем модели:
   - классические регрессоры;
   - нейросети;
   - модели внутри кластеров.
7. Сравниваем метрики (MSE, RMSE, MAE, R2) и делаем выводы.

## Запуск CLI-приложения

На Windows чаще используется команда `python`, а не `python3`.

### Режим 1: запуск без входного JSON

Это самый простой режим. Файл `input.json` не нужен. Приложение само возьмет медианные значения признаков из датасета.

- Прогноз модуля упругости:

  `python app.py --target modulus`

- Прогноз прочности:

  `python app.py --target strength`

- Рекомендация соотношения матрица-наполнитель:

  `python app.py --target ratio`

### Режим 2: запуск с готовым примером JSON

В проекте есть файл `input_example.json`. Его можно использовать как пример:

`python app.py --target strength --input-json input_example.json`

### Режим 3: запуск со своим JSON

Создай рядом с `app.py` файл, например `input.json`, и заполни его значениями признаков:

```json
{
  "Соотношение матрица-наполнитель": 2.5,
  "Плотность, кг/м3": 2000,
  "модуль упругости, ГПа": 700,
  "Количество отвердителя, м.%": 30,
  "Содержание эпоксидных групп,%_2": 22,
  "Температура вспышки, С_2": 100,
  "Поверхностная плотность, г/м2": 210,
  "Потребление смолы, г/м2": 220,
  "Угол нашивки, град": 0,
  "Шаг нашивки": 4,
  "Плотность нашивки": 57
}
```

После этого запусти:

`python app.py --target strength --input-json input.json`

Важно: если указать `--input-json input.json`, файл `input.json` действительно должен существовать в текущей папке.

### Все варианты `--target`

- `modulus` — прогноз `Модуль упругости при растяжении, ГПа`.
- `strength` — прогноз `Прочность при растяжении, МПа`.
- `ratio` — рекомендация `Соотношение матрица-наполнитель`.

### Короткая памятка по командам

- `python app.py --target modulus`
- `python app.py --target strength`
- `python app.py --target ratio`
- `python app.py --target strength --input-json input_example.json`

Если `input.json` не передан, приложение использует медианные значения признаков.

## Частые ошибки и как исправить

- **Ошибка `FileNotFoundError` при чтении `X_bp.xlsx` или `X_nup.xlsx`**  
  Проверь, что запуск идет из корня проекта и папка `hw_data_composite` лежит рядом с ноутбуками.

- **Ошибка `FileNotFoundError: input.json`**  
  Ты запустил приложение с параметром `--input-json input.json`, но такого файла нет. Запусти без `--input-json` или используй готовый пример: `python app.py --target strength --input-json input_example.json`.

- **Ошибка `ModuleNotFoundError`**  
  Не хватает библиотек. На Windows выполни: `python -m pip install -r requirements.txt`.

- **Ноутбук выполняется очень долго**  
  Уменьши число эпох в нейросети (например, со 100/300 до 30/50) для чернового прогона.

- **Метрики отличаются при повторном запуске**  
  Это ожидаемо при случайном разбиении train/test. Для стабильности можно зафиксировать `random_state`.

- **После кластеризации качество хуже, чем ожидалось**  
  Это нормальная ситуация: кластеризация не всегда улучшает регрессию. В выводах важно честно сравнить подходы и указать, где результат лучше.

## Короткие ответы для защиты

- **Почему используете два файла, а не один?**  
  В исходных данных признаки разделены на две таблицы. Мы объединяем их по строкам в единый датафрейм, чтобы использовать всю информацию.

- **Почему не удаляли все выбросы?**  
  Часть крайних значений похожа на реальные технологические режимы. Массовое удаление могло бы ухудшить репрезентативность выборки.

- **Почему результаты у моделей разные?**  
  Модели имеют разную сложность и по-разному улавливают нелинейные зависимости. Поэтому качество по MSE/MAE/R2 отличается.

- **Зачем нужна кластеризация?**  
  Чтобы проверить гипотезу о разных типах композитов в данных и попробовать строить отдельные модели внутри групп.

- **Как проверяли качество?**  
  По стандартным метрикам регрессии: MSE, RMSE, MAE, R2, отдельно на train и test.
